Table-Driven Tests

The canonical Go testing pattern — and a chance to verify that your dice.go actually implements the modifier formula correctly.

You wrote dice.go. Now prove it works. In Go, the standard library ships a testing package and a built-in test runner. No pytest, no unittest — it's part of the toolchain. This lesson covers the pattern Go programmers reach for almost universally: the table-driven test.

Review Two Things to Note in Your Code

Before adding tests, two things in your dice.go are worth flagging — not wrong, but worth understanding.

1. The blank identifier _

In AbilityCheck, you wrote:

roll, _ := Roll(20)

The _ is Go's blank identifier — it discards the value. Here you're silently ignoring the error from Roll(20). Go makes this a conscious choice: you have to explicitly reach for _, you can't just forget to handle a return value.

For Roll(20) specifically, the error path is unreachable — you're passing the literal 20, which will always pass the guard. So discarding it is defensible. But in production bot code you'd want to propagate it anyway so callers can log it. We'll revisit this once we have a logger. For now, note what you did and why it's acceptable here.

2. One-sided dice

Your guard is sides <= 0. A 1-sided die (Roll(1)) passes: rand.Intn(1) always returns 0, so Roll(1) always returns 1. Technically valid, but D&D doesn't have 1-sided dice. The lesson used sides < 2 to prevent it. This is a micro-decision — either is fine — but it's exactly the kind of thing a test would catch and document.

Go Testing Fundamentals

The test file

Go's test runner looks for files named *_test.go. These files are compiled and run with go test but ignored in regular builds. Convention: the test file for dice.go lives in the same directory as dice_test.go and belongs to the same package.

Python (pytest)
# test_dice.py
import pytest
from dice import ability_modifier

def test_modifier():
    assert ability_modifier(10) == 0
Go
// dice_test.go
package dice

import "testing"

func TestAbilityModifier(t *testing.T) {
    got := AbilityModifier(10)
    if got != 0 {
        t.Errorf("score 10: got %d, want 0", got)
    }
}

Test functions take exactly one argument: *testing.T. The name must start with Test. t.Errorf marks the test as failed and logs a message — the test keeps running. Use t.Fatalf if you want to stop immediately.

Running tests

$ go test ./pkg/dice/ ok github.com/tjlsmith0831/dnd-go-agent/pkg/dice 0.003s $ go test -v ./pkg/dice/ === RUN TestAbilityModifier --- PASS: TestAbilityModifier (0.00s) PASS $ go test ./... # runs all packages

The table-driven pattern

One test function per case doesn't scale. Go's idiomatic answer is to define a slice of test cases — a table — and range over it. This is so common it's practically a language convention.[Go Wiki]

func TestAbilityModifier(t *testing.T) {
    tests := []struct {
        score int
        want  int
    }{
        {1, -5},
        {8, -1},
        {9, -1},  // the floor-division case — would be 0 with plain / in Go
        {10, 0},
        {11, 0},
        {14, 2},
        {20, 5},
    }

    for _, tc := range tests {
        got := AbilityModifier(tc.score)
        if got != tc.want {
            t.Errorf("score %d: got %d, want %d", tc.score, got, tc.want)
        }
    }
}

The struct type is defined inline — []struct{ ... }{ ... }. You'll see this constantly. It keeps the test data co-located with the test, with no need for a named type.

Named subtests with t.Run

Add a name field and wrap in t.Run and each case becomes an individually addressable, runnable, nameable subtest:

func TestAbilityModifier(t *testing.T) {
    tests := []struct {
        name  string
        score int
        want  int
    }{
        {"min score", 1, -5},
        {"odd below 10", 9, -1},
        {"neutral", 10, 0},
        {"odd above 10", 11, 0},
        {"strong fighter", 18, 4},
        {"max standard", 20, 5},
    }

    for _, tc := range tests {
        t.Run(tc.name, func(t *testing.T) {
            got := AbilityModifier(tc.score)
            if got != tc.want {
                t.Errorf("got %d, want %d", got, tc.want)
            }
        })
    }
}

Now you can run a specific case: go test -run TestAbilityModifier/odd_below_10. The slash separates the test function from the subtest name; spaces become underscores.

Testing Roll

Random output is harder to test than deterministic functions. You can't assert the exact value. Instead, test the contract: that the output is always within range, and that invalid input returns an error.

func TestRoll(t *testing.T) {
    t.Run("valid range", func(t *testing.T) {
        for range 1000 {
            got, err := Roll(20)
            if err != nil {
                t.Fatalf("unexpected error: %v", err)
            }
            if got < 1 || got > 20 {
                t.Errorf("Roll(20) = %d, want 1–20", got)
            }
        }
    })

    t.Run("invalid sides", func(t *testing.T) {
        _, err := Roll(0)
        if err == nil {
            t.Error("expected error for sides=0, got nil")
        }
    })
}

The for range 1000 syntax (Go 1.22+) loops 1000 times without a loop variable — you just need the iterations. Running many samples gives statistical confidence without fixing the seed.

Go idiom: t.Error vs t.Fatal
t.Error/t.Errorf — marks failed, continues running.
t.Fatal/t.Fatalf — marks failed, stops the current test immediately.
Use Fatal when subsequent assertions would be nonsensical if the first one failed (e.g., nil pointer dereference). Otherwise prefer Error so you see all failures at once.

The complete test file

Create pkg/dice/dice_test.go:

package dice

import "testing"

func TestAbilityModifier(t *testing.T) {
	tests := []struct {
		name  string
		score int
		want  int
	}{
		{"min score", 1, -5},
		{"odd below ten", 9, -1},  // floor-division case
		{"neutral score", 10, 0},
		{"odd above ten", 11, 0},
		{"strong fighter", 18, 4},
		{"max standard", 20, 5},
	}
	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			got := AbilityModifier(tc.score)
			if got != tc.want {
				t.Errorf("score %d: got %d, want %d", tc.score, got, tc.want)
			}
		})
	}
}

func TestRoll(t *testing.T) {
	t.Run("valid range", func(t *testing.T) {
		for range 1000 {
			got, err := Roll(20)
			if err != nil {
				t.Fatalf("unexpected error: %v", err)
			}
			if got < 1 || got > 20 {
				t.Errorf("Roll(20) = %d, want 1–20", got)
			}
		}
	})
	t.Run("invalid sides", func(t *testing.T) {
		_, err := Roll(0)
		if err == nil {
			t.Error("expected error for sides=0, got nil")
		}
	})
}
