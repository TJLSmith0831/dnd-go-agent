Methods & Interfaces

How Go replaces class hierarchies — and how you'll define what the bot needs from any combatant, whether PC or monster.

In Python, behavior lives inside a class. In Go, behavior is attached to a type via methods, and types are grouped by capability via interfaces. These two mechanisms replace inheritance entirely. By the end of this lesson you'll have a Combatant interface your bot can use to run combat without caring whether it's talking to a player character or a goblin.

Go Methods on Structs

A method is a function with a receiver — a type argument that comes before the function name. The receiver is what connects the function to the type:

Python
class CheckResult:
    def is_critical(self) -> bool:
        return self.roll == 20
Go
func (r CheckResult) IsCritical() bool {
    return r.Roll == 20
}

The receiver (r CheckResult) sits between func and the method name. r is just a local name for the value — you'll commonly see single letters or short abbreviations. Call it like Python: result.IsCritical().

Value vs Pointer Receivers

This is where Go diverges from Python most sharply. Receivers come in two forms:

Value receiver
(r CheckResult)

Gets a copy of the struct. The original is untouched. Use when the method only reads state.
Pointer receiver
(r *CheckResult)

Gets a pointer to the original. Can modify it. Use when the method mutates state, or when the struct is large enough that copying is wasteful. 
// Value receiver — reads only, gets a copy
func (r CheckResult) IsCritical() bool {
    return r.Roll == 20
}

// Value receiver — formats for display
func (r CheckResult) String() string {
    outcome := "miss"
    if r.Success {
        outcome = "hit"
    }
    return fmt.Sprintf("roll %d + %d = %d vs DC %d (%s)",
        r.Roll, r.Modifier, r.Total, r.DC, outcome)
}
Consistency rule: if any method on a type uses a pointer receiver, all of them should. Mixing value and pointer receivers on the same type breaks interface satisfaction in subtle, confusing ways. Pick one and stick to it. For types that hold mutable state (like a combatant's HP), use pointer receivers throughout.

The fmt.Stringer Interface

The String() string method above is not arbitrary — it satisfies fmt.Stringer, a standard library interface. Any type with a String() string method will automatically be formatted by fmt.Println, fmt.Sprintf, and friends using that method. You didn't register anything; the compiler figured it out.

result, _ := dice.AbilityCheck(14, 15)
fmt.Println(result)  // → "roll 9 + 2 = 11 vs DC 15 (miss)"

Go Interfaces

An interface is a named set of method signatures. Any type that has all those methods automatically satisfies the interface — no declaration, no registration, no implements keyword.[Tour of Go]

Python (explicit)
from abc import ABC, abstractmethod

class Combatant(ABC):
    @abstractmethod
    def name(self) -> str: ...
    @abstractmethod
    def current_hp(self) -> int: ...
Go (implicit)
type Combatant interface {
    Name()      string
    CurrentHP() int
}

Any type with Name() string and CurrentHP() int methods satisfies Combatant — even types defined in packages that have never heard of this interface. This is called structural typing. It's what makes Go interfaces composable without coupling.

The Combatant Interface for Our Bot

Based on what the bot must track (from lesson 3), here's the interface that both player characters and monsters need to satisfy:

// Combatant is anything that can participate in D&D 5e combat.
type Combatant interface {
    Name()           string
    CurrentHP()      int
    MaxHP()          int
    AC()             int
    InitiativeBonus() int   // DEX modifier
    IsAlive()        bool
    TakeDamage(amount int)
}

Two Types, One Interface

Now define two concrete types. Neither mentions Combatant. The compiler will verify satisfaction at the point of use:

type Monster struct {
    name      string
    currentHP int
    maxHP     int
    ac        int
    dexMod    int
}

func (m *Monster) Name()            string { return m.name }
func (m *Monster) CurrentHP()       int    { return m.currentHP }
func (m *Monster) MaxHP()           int    { return m.maxHP }
func (m *Monster) AC()              int    { return m.ac }
func (m *Monster) InitiativeBonus() int    { return m.dexMod }
func (m *Monster) IsAlive()         bool   { return m.currentHP > 0 }
func (m *Monster) TakeDamage(amount int) {
    m.currentHP -= amount
    if m.currentHP < 0 {
        m.currentHP = 0
    }
}

A PlayerCharacter type would look similar but include death save tracking. The combat engine doesn't need to know which it's dealing with:

// The engine works on the interface. Monster and PlayerCharacter
// can be mixed freely in the same slice.
func RunInitiative(combatants []Combatant) {
    for _, c := range combatants {
        roll, _ := dice.Roll(20)
        score := roll + c.InitiativeBonus()
        fmt.Printf("%s rolls initiative: %d\n", c.Name(), score)
    }
}
The design insight: the interface is defined by the consumer (the combat engine), not by the types that satisfy it. Monster doesn't know it satisfies Combatant. This is the opposite of Python's ABC pattern, and it means you can make any existing type satisfy a new interface without touching it.

Gotcha Pointer Receivers and Interface Satisfaction

This is the most common interface mistake in Go. If a method is defined on *Monster (pointer receiver), then *Monster satisfies the interface — but plain Monster does not:

var c Combatant

goblin := Monster{name: "Goblin", currentHP: 7, maxHP: 7, ac: 15}

c = &goblin  // ✓ *Monster satisfies Combatant
c = goblin   // ✗ Monster does not — compile error

The rule: pointer to type satisfies interfaces defined with pointer receivers; value of type does not. If you see a compile error like "Monster does not implement Combatant (TakeDamage method has pointer receiver)", this is why. Use &goblin.
