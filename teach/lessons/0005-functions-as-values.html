Functions as Values

Fixing RunInitiative the TDD way — and learning how Go treats functions as first-class citizens.

RunInitiative currently just prints. The combat engine needs it to return combatants in initiative order — highest score first, ties broken by DEX modifier. We'll build this the right way: test first.

TDD Write the Test First

Before touching the implementation, write the test that defines what "sorted initiative" means. This test will fail immediately — that's the point.

Add this to pkg/combat/combatant_test.go:

func TestRollInitiative(t *testing.T) {
    combatants := []Combatant{
        &Monster{name: "Goblin", currentHP: 7, maxHP: 7, ac: 15, dexMod: 2},
        &Monster{name: "Ogre", currentHP: 59, maxHP: 59, ac: 11, dexMod: -1},
        &PlayerCharacter{name: "Aria", currentHP: 28, maxHP: 28, ac: 16, dexMod: 3},
    }

    ordered := RollInitiative(combatants)

    if len(ordered) != len(combatants) {
        t.Fatalf("got %d entries, want %d", len(ordered), len(combatants))
    }
    for i := 1; i < len(ordered); i++ {
        if ordered[i].Score > ordered[i-1].Score {
            t.Errorf("not sorted: index %d (score %d) > index %d (score %d)",
                i, ordered[i].Score, i-1, ordered[i-1].Score)
        }
    }
}

Go Functions as First-Class Values

In Go, functions are values. You can assign them to variables, pass them as arguments, and return them from other functions — exactly like Python.[Tour of Go]

Python
def double(x):
    return x * 2

apply = double
apply(5)  # → 10

def run(fn, x):
    return fn(x)

run(double, 5)  # → 10
Go
func double(x int) int { return x * 2 }

apply := double
apply(5)  // → 10

func run(fn func(int) int, x int) int {
    return fn(x)
}

run(double, 5)  // → 10

The type of a function is written as func(paramTypes) returnType. A function that takes an int and returns an int has type func(int) int. A function that takes two ints and returns a bool has type func(int, int) bool.

Closures

A closure is a function that captures variables from the scope where it was defined — not passed in as arguments, just referenced directly:

Python
def make_adder(n):
    def add(x):
        return x + n  # closes over n
    return add

add5 = make_adder(5)
add5(3)  # → 8
Go
func makeAdder(n int) func(int) int {
    return func(x int) int {
        return x + n  // closes over n
    }
}

add5 := makeAdder(5)
add5(3)  // → 8

The anonymous function inside makeAdder captures n from its surrounding scope. That captured variable lives as long as the closure does, even after makeAdder returns. You'll use this constantly with sort.Slice.

sort.Slice

The standard library's sort.Slice takes a slice and a less function — a closure that tells it how to compare two elements by index:

import "sort"

scores := []int{4, 12, 7, 19, 3}

// Sort ascending
sort.Slice(scores, func(i, j int) bool {
    return scores[i] < scores[j]
})
// scores → [3, 4, 7, 12, 19]

// Sort descending — flip the comparison
sort.Slice(scores, func(i, j int) bool {
    return scores[i] > scores[j]
})
// scores → [19, 12, 7, 4, 3]

The closure closes over scores — it references the slice directly rather than receiving it as an argument. sort.Slice only calls the function with indices; the closure handles the rest.

Go D&D Implementing RollInitiative

The function needs a return type. A combatant's initiative score isn't part of the Combatant interface — it's the result of a die roll that happens once at the start of combat. A small struct holds both together:

// InitiativeEntry pairs a combatant with their rolled initiative score.
type InitiativeEntry struct {
    Combatant Combatant
    Score     int
}

Now the function: roll for each combatant, build the slice, sort it descending.

// RollInitiative rolls initiative for all combatants and returns them
// sorted from highest score to lowest.
func RollInitiative(combatants []Combatant) ([]InitiativeEntry, error) {
    entries := make([]InitiativeEntry, len(combatants))

    for i, c := range combatants {
        roll, err := dice.Roll(20)
        if err != nil {
            return nil, err
        }
        entries[i] = InitiativeEntry{
            Combatant: c,
            Score:     roll + c.InitiativeBonus(),
        }
    }

    sort.Slice(entries, func(i, j int) bool {
        return entries[i].Score > entries[j].Score  // descending
    })

    return entries, nil
}
New built-in: make
make([]InitiativeEntry, len(combatants)) allocates a slice of a specific length, with all elements zero-valued. It's the idiomatic way to pre-allocate a slice when you know the size upfront — more efficient than starting empty and appending. make works on slices, maps, and channels; it's not for structs or arrays.

Gotcha Closures Capture Variables, Not Values

The most common closure mistake in Go — especially in loops. This prints 3, 3, 3 instead of 1, 2, 3:

// WRONG: all closures capture the same i variable
funcs := []func(){}
for i := 1; i <= 3; i++ {
    funcs = append(funcs, func() { fmt.Println(i) })
}
for _, f := range funcs { f() }  // prints 4, 4, 4 (i's final value)

// RIGHT: capture the value by shadowing i in the inner scope
for i := 1; i <= 3; i++ {
    i := i  // new variable shadows the loop variable
    funcs = append(funcs, func() { fmt.Println(i) })
}
// prints 1, 2, 3

Go 1.22+ (which you're on) fixed this for for loop variables specifically — each iteration gets its own copy of i. But the pattern appears in other contexts too. Know it.
