Type Assertions & Type Switches

Asking "what are you, really?" — and building the combat loop's 0 HP handler the TDD way.

The combat loop has a problem: when a combatant reaches 0 HP, it needs to do different things depending on whether it's a *PlayerCharacter or a *Monster. The Combatant interface can't express this difference — and it shouldn't. Type assertions are Go's mechanism for reaching past an interface to the concrete type underneath.

TDD Write the Test First

We want a function HandleZeroHP(c Combatant) that returns a string describing what happened. Monsters die. PCs enter death saves. Write the test before writing a line of implementation.

Add to pkg/combat/combatant_test.go:

func TestHandleZeroHP(t *testing.T) {
    t.Run("monster dies at zero HP", func(t *testing.T) {
        m := &Monster{name: "Goblin", currentHP: 0, maxHP: 7}
        outcome := HandleZeroHP(m)
        if outcome != OutcomeDead {
            t.Errorf("got %v, want OutcomeDead", outcome)
        }
    })

    t.Run("PC at zero HP makes a death save", func(t *testing.T) {
        pc := &PlayerCharacter{name: "Aria", currentHP: 0, maxHP: 28}
        outcome := HandleZeroHP(pc)
        if outcome != OutcomeDeathSave && outcome != OutcomeStabilized &&
            outcome != OutcomeDead && outcome != OutcomeAwakened {
            t.Errorf("unexpected outcome for PC: %v", outcome)
        }
    })

    t.Run("non-zero HP combatant is unaffected", func(t *testing.T) {
        m := &Monster{name: "Ogre", currentHP: 30, maxHP: 59}
        outcome := HandleZeroHP(m)
        if outcome != OutcomeAlive {
            t.Errorf("got %v, want OutcomeAlive", outcome)
        }
    })
}

Go Type Assertions

A type assertion extracts the concrete value from an interface variable. There are two forms:[Tour of Go]

// Single-value form — panics if c is not *PlayerCharacter
pc := c.(*PlayerCharacter)

// Two-value form — safe, ok is false if the assertion fails
pc, ok := c.(*PlayerCharacter)
if ok {
    // pc is a *PlayerCharacter here
}
Always use the two-value form unless you are certain of the type and a panic is acceptable. The single-value form is rare in production code — it's mostly used in tests or when a wrong type is truly a programming error that should crash immediately.
Python
if isinstance(c, PlayerCharacter):
    pc = c
    pc.death_saving_throw()
Go
if pc, ok := c.(*PlayerCharacter); ok {
    pc.DeathSavingThrow()
}

The Go form declares pc and ok in the same if initializer — both are scoped to the if block. pc is typed as *PlayerCharacter inside, so you can call any method on it, not just those on the Combatant interface.

Type Switches

When you need to branch on more than one possible type, a type switch is cleaner than chaining assertions:

switch v := c.(type) {
case *PlayerCharacter:
    // v is *PlayerCharacter here
    v.DeathSavingThrow()
case *Monster:
    // v is *Monster here
    fmt.Printf("%s is slain\n", v.Name())
default:
    // v is Combatant — the original interface type
}

Inside each case, v is automatically typed as the matched concrete type. No manual cast needed.

Go D&D Implementing HandleZeroHP

First, define the outcome type. Using a named string type (rather than plain strings) means the compiler catches typos at the call site:

type ZeroHPOutcome string

const (
    OutcomeAlive     ZeroHPOutcome = "alive"
    OutcomeDead      ZeroHPOutcome = "dead"
    OutcomeDeathSave ZeroHPOutcome = "death_save"
    OutcomeStabilized ZeroHPOutcome = "stabilized"
    OutcomeAwakened  ZeroHPOutcome = "awakened"
)
Named types vs type aliases
type ZeroHPOutcome string creates a new type — you cannot accidentally assign a plain string to a ZeroHPOutcome variable without an explicit conversion. This is intentional: it prevents passing "ded" where OutcomeDead is expected. It's a lightweight alternative to an enum.

Now the function:

func HandleZeroHP(c Combatant) ZeroHPOutcome {
    if c.CurrentHP() > 0 {
        return OutcomeAlive
    }

    switch v := c.(type) {
    case *PlayerCharacter:
        result, err := v.DeathSavingThrow()
        if err != nil {
            return OutcomeDead  // shouldn't happen — HP check above guards it
        }
        switch {
        case result.Awakened:
            return OutcomeAwakened
        case result.Dead:
            return OutcomeDead
        case result.Stabilized:
            return OutcomeStabilized
        default:
            return OutcomeDeathSave
        }
    default:
        // Monsters and anything else: dead at 0 HP
        return OutcomeDead
    }
}

Design smell When type assertions mean the interface is wrong

Type assertions are legitimate here because death saves are genuinely a PC-only mechanic — no interface change could cleanly express it. But watch for this anti-pattern:

// Smell: type-asserting just to call a method that ALL combatants could have
if m, ok := c.(*Monster); ok {
    m.Name()  // Name() is already on the Combatant interface — no assertion needed
}

If you find yourself type-asserting to call a method that conceptually applies to all combatants, the right fix is adding that method to the interface. Type assertions are for behavior that is genuinely specific to one concrete type.
