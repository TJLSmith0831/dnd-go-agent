Goroutines & Channels

How Go handles concurrency — and why it's the right model for a bot that runs dozens of campaign sessions at once.

In Python you reach for asyncio or threads when you need to do two things at once. Go has a different answer: goroutines and channels. These aren't a library bolt-on — they're built into the language. Understanding them is the biggest conceptual jump from Python to Go, and it's the foundation of everything the bot does concurrently: handling multiple Discord guilds, routing agent messages, processing dice rolls while streaming a narrative response.

Go Goroutines

A goroutine is a function running concurrently with the rest of your program. You launch one with the go keyword:

go someFunction()

That's it. The function starts immediately in the background. Your current code keeps running — it doesn't wait.

How goroutines differ from threads

OS threads are expensive: each one reserves ~1MB of stack space and requires a kernel context switch. Goroutines start with a ~2KB stack that grows as needed and are multiplexed onto a small pool of OS threads by the Go runtime scheduler. You can run tens of thousands of goroutines in a single process without issue — the bot can maintain one goroutine per active campaign session with no sweat.

Python analogy: goroutines are closer to asyncio coroutines than to threads — they're cooperative and scheduled by the runtime. But unlike asyncio, you don't need await everywhere. Any blocking call (channel read, disk I/O) just yields the goroutine automatically. The scheduler is preemptive.

Goroutine lifecycle

A goroutine exits when its function returns. There's no handle you can join or cancel directly — you signal it through a channel (covered below). This matters: a goroutine that's blocked waiting on a channel with no sender will leak forever. The Go race detector (go test -race) catches many of these.

Go Channels

Goroutines communicate through channels. A channel is a typed pipe: you send a value in from one end and receive it out the other. The Go mantra is:

"Don't communicate by sharing memory; share memory by communicating."
Instead of a shared variable protected by a mutex, pass values through a channel. One goroutine owns the value; it hands it off via the channel. No lock needed.

Creating and using a channel

// make(chan T) creates an unbuffered channel of type T
ch := make(chan string)

// send: ch <- value   (blocks until a receiver is ready)
ch <- "hello"

// receive: value := <-ch   (blocks until a sender sends)
msg := <-ch

Unbuffered channels block on both ends. The sender waits until a receiver is ready; the receiver waits until a sender sends. This synchronization is the whole point — it's a rendezvous, not a queue.

Directional channels

You can restrict a channel to send-only or receive-only. This is idiomatic in function signatures — it makes intent explicit and lets the compiler enforce it:

// send-only: can only write to this channel
func produce(out chan<- string) { out <- "roll" }

// receive-only: can only read from this channel
func consume(in <-chan string) { msg := <-in; _ = msg }

A bidirectional chan string converts implicitly to either directional form — you make it at the call site and pass it in restricted.

Ranging over a channel

A for range loop on a channel reads values until the channel is closed. This is the standard agent loop pattern:

for msg := range in {
    // process each message as it arrives
    out <- process(msg)
}
// loop exits only when in is closed

The loop blocks between messages — the goroutine is suspended until the next value arrives. Closing the channel from the sender side signals "no more values." Closing from the receiver side is a panic — only the sender closes.

Go Select

select is like a switch for channel operations. It blocks until one of its cases is ready, then executes that case. If multiple are ready, it picks one at random:

select {
case msg := <-playerMsg:
    handlePlayerMessage(msg)
case <-done:
    return // shutdown signal received
}

The done channel pattern is the standard way to cancel a goroutine: the parent closes a chan struct{} (zero-size type, costs nothing), and the goroutine's select picks it up and returns.

How this maps to the bot:
One goroutine per Discord guild — it loops over an incoming message channel with for range and a done channel in select. The orchestrator sends player messages in; the agent loop reads them out and sends LLM responses back through an outbound channel. Campaign sessions start and stop by opening and closing channels.

Go TDD: Build a Session Runner

You're going to build pkg/session — a goroutine that processes incoming game messages and sends results out. This is the skeleton of the bot's per-guild loop.

Step 1 — write the failing test first. Create pkg/session/session_test.go:

Red — test that doesn't compile yet
package session

import "testing"

func TestRun(t *testing.T) {
    in   := make(chan string)
    out  := make(chan string)
    done := make(chan struct{})

    go Run(in, out, done)

    in <- "attack goblin"
    got := <-out
    if got != "attack goblin" {
        t.Errorf("got %q, want %q", got, "attack goblin")
    }

    close(done)
}

Run go test ./pkg/session/ — it fails to compile because Run doesn't exist. That's the red state you want.

Step 2 — implement to green. Create pkg/session/session.go:

Green — minimal implementation
package session

func Run(in <-chan string, out chan<- string, done <-chan struct{}) {
    for {
        select {
        case msg := <-in:
            out <- msg
        case <-done:
            return
        }
    }
}

Run go test ./pkg/session/ — it should pass. Notice:

  • in is <-chan string — receive-only, Run can't accidentally send to it
  • out is chan<- string — send-only, Run can't accidentally read from it
  • done is <-chan struct{} — receive-only; struct{} costs zero bytes
  • The select races in against done — whichever is ready first wins
  • The test calls go Run(...) — Run is the goroutine, not the test
If the test hangs: you have a deadlock. Either Run is blocked waiting to send and nothing's receiving, or the test is blocked waiting to receive and Run hasn't sent. Add t.Parallel() and a timeout context if you want to catch hangs automatically.
