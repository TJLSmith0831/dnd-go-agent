Lesson 0012 · D&D 5e + Go

Conditions & iota

The status effects that flip attack rolls and restrict actions — and Go's bit-flag pattern that tracks them all in a single integer.

A dragon breathes fire and three party members fall Prone. The wizard casts Hold Person and the bandit captain is Paralyzed. The rogue drinks poison by mistake and is now Poisoned. Each condition changes the rules of combat for that character — which rolls have disadvantage, what actions are forbidden, whether hits are automatic crits. The Referee agent has to track all of this per combatant and apply it at the right moment.

D&D 5e The Conditions

There are 15 official conditions in 5e. These are the eight most common in play:

ConditionKey mechanical effects
BlindedAuto-fail sight checks. Your attacks: disadvantage. Attacks against you: advantage.
FrightenedDisadvantage on attacks and ability checks while source is visible. Can't move closer to source.
IncapacitatedCan't take actions or reactions. Many conditions include this.
ParalyzedIncapacitated + can't move or speak. Auto-fail STR/DEX saves. Attacks against: advantage. Melee hits within 5 ft: auto-crit.
PoisonedDisadvantage on attack rolls and ability checks.
ProneYour attacks: disadvantage. Melee attacks against: advantage. Ranged attacks against: disadvantage. Costs half movement to stand.
RestrainedSpeed 0. Your attacks: disadvantage. Your DEX saves: disadvantage. Attacks against: advantage.
StunnedIncapacitated + can't move. Auto-fail STR/DEX saves. Attacks against: advantage.

Conditions that cascade

Paralyzed includes Incapacitated. So does Stunned. If you're tracking conditions as a set, a Paralyzed character is also effectively Incapacitated — you don't add it separately, but any check for Incapacitated must match. The cleanest model: the Referee checks each condition independently when computing modifiers, and Paralyzed's definition lists all its effects explicitly.

BG3 note: the red/orange/purple icons that stack on a character portrait during combat are all conditions. When a character is Stunned, BG3 greys out their action button — that's Incapacitated enforced by the engine. Your bot does the same thing in code.

D&D 5e How Conditions Affect Rolls

Most conditions apply to the d20 roll via Advantage or Disadvantage (recall: roll two d20s, take the higher or lower). The bot's resolution path gains a step:

1. Is the attacker's condition granting disadvantage on this attack? 2. Is the defender's condition granting advantage to the attacker? 3. Do both apply? → they cancel (any advantage + any disadvantage = straight roll) 4. Roll accordingly, then check for auto-crit conditions (Paralyzed, Stunned)

This is why conditions are tracked as a set, not a list of modifiers. You query the set at resolution time: "does the attacker have any condition that penalises attacks?" rather than precomputing a number.

Go iota and Bit Flags

Eight conditions that can be present or absent in any combination — this is a set of boolean flags. You could use a map[string]bool, but Go has a more compact and efficient pattern: bit flags with iota.

iota is a counter that resets to 0 at the start of each const block and increments by 1 for each constant. Used with bit-shifting, it gives each constant its own unique bit:

type Condition uint16

const (
    Blinded       Condition = 1 << iota // 0000_0001 = 1
    Frightened                             // 0000_0010 = 2
    Incapacitated                          // 0000_0100 = 4
    Paralyzed                              // 0000_1000 = 8
    Poisoned                               // 0001_0000 = 16
    Prone                                  // 0010_0000 = 32
    Restrained                             // 0100_0000 = 64
    Stunned                                // 1000_0000 = 128
)

Each constant occupies exactly one bit. A uint16 can hold all 16 possible conditions in 2 bytes.

Bitwise operations on the set

type ConditionSet struct{ flags Condition }

// Apply sets a bit:  OR with the condition's bit
func (cs *ConditionSet) Apply(c Condition)  { cs.flags |= c }

// Remove clears a bit: AND-NOT (&^) with the condition's bit
func (cs *ConditionSet) Remove(c Condition) { cs.flags &^= c }

// Has checks a bit:  AND — non-zero means the bit is set
func (cs *ConditionSet) Has(c Condition) bool { return cs.flags&c != 0 }

&^ is Go's bit-clear (AND-NOT) operator. Python doesn't have it — the equivalent is flags & ~condition. It zeroes the bits in the left operand wherever the right operand has a 1.

Checking multiple conditions at once

// Does this combatant have disadvantage on their attack roll?
func (cs *ConditionSet) AttackDisadvantage() bool {
    return cs.Has(Blinded) || cs.Has(Frightened) ||
        cs.Has(Poisoned) || cs.Has(Prone) || cs.Has(Restrained)
}

// Do attacks against this combatant have advantage?
func (cs *ConditionSet) GrantsAdvantageToAttackers() bool {
    return cs.Has(Blinded) || cs.Has(Paralyzed) ||
        cs.Has(Restrained) || cs.Has(Stunned)
}

Red — conditions_test.go
package combat

import "testing"

func TestConditionSet(t *testing.T) {
    t.Run("Apply and Has round-trip", func(t *testing.T) {
        var cs ConditionSet
        cs.Apply(Poisoned)
        if !cs.Has(Poisoned) {
            t.Error("expected Poisoned after Apply")
        }
    })

    t.Run("Remove clears the condition", func(t *testing.T) {
        var cs ConditionSet
        cs.Apply(Prone)
        cs.Remove(Prone)
        if cs.Has(Prone) {
            t.Error("expected Prone cleared after Remove")
        }
    })

    t.Run("Remove does not affect other conditions", func(t *testing.T) {
        var cs ConditionSet
        cs.Apply(Poisoned)
        cs.Apply(Prone)
        cs.Remove(Prone)
        if !cs.Has(Poisoned) {
            t.Error("Remove(Prone) should not clear Poisoned")
        }
    })

    t.Run("Poisoned gives attack disadvantage", func(t *testing.T) {
        var cs ConditionSet
        cs.Apply(Poisoned)
        if !cs.AttackDisadvantage() {
            t.Error("Poisoned should give attack disadvantage")
        }
    })

    t.Run("Paralyzed grants advantage to attackers", func(t *testing.T) {
        var cs ConditionSet
        cs.Apply(Paralyzed)
        if !cs.GrantsAdvantageToAttackers() {
            t.Error("Paralyzed should grant advantage to attackers")
        }
    })
}

package combat

type Condition uint16

const (
    Blinded       Condition = 1 << iota
    Frightened
    Incapacitated
    Paralyzed
    Poisoned
    Prone
    Restrained
    Stunned
)

type ConditionSet struct{ flags Condition }

func (cs *ConditionSet) Apply(c Condition)       { cs.flags |= c }
func (cs *ConditionSet) Remove(c Condition)      { cs.flags &^= c }
func (cs *ConditionSet) Has(c Condition) bool   { return cs.flags&c != 0 }

func (cs *ConditionSet) AttackDisadvantage() bool {
    return cs.Has(Blinded) || cs.Has(Frightened) ||
        cs.Has(Poisoned) || cs.Has(Prone) || cs.Has(Restrained)
}

func (cs *ConditionSet) GrantsAdvantageToAttackers() bool {
    return cs.Has(Blinded) || cs.Has(Paralyzed) ||
        cs.Has(Restrained) || cs.Has(Stunned)
}

Note var cs ConditionSet in the tests — no need for make or a constructor. A zero-value ConditionSet has flags == 0, meaning no conditions. That's exactly right.
