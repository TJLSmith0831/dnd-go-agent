Lesson 0013 · Go

Postgres & pgx

The bot can hold a conversation — but it forgets everything when it restarts. This lesson adds the memory layer.

Run the bot right now and play a session. Quit. Run it again. The Narrator has no idea the session happened. For a one-shot that's fine, but a campaign needs continuity — the dragon was wounded last week, the tavern burned down, the NPC the party befriended is still alive. That state lives in Postgres. This lesson wires it in.

What the bot gains from this lesson: a Store that saves every message exchanged in a session and reloads the history on reconnect. The Narrator can now reference what happened previously without it being in the current context window.

Bot The Schema

Two tables. Keep it minimal — you can always add columns later, but you can't un-ship a bad schema.

-- One row per campaign/guild. guild_id ties to the Discord server.
CREATE TABLE sessions (
    id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    guild_id   TEXT        NOT NULL UNIQUE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- One row per message exchanged. Role identifies who spoke.
CREATE TABLE events (
    id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id UUID        NOT NULL REFERENCES sessions(id),
    role       TEXT        NOT NULL CHECK (role IN ('player', 'narrator', 'referee', 'npc')),
    content    TEXT        NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

The events table is the campaign journal. Every player action and every Narrator response is a row. When the bot reconnects, it queries the last N events and injects them as the LLM's message history — the Narrator "remembers" because the history is reconstructed from the database.

Go pgx vs database/sql

Go's standard library has database/sql — a generic interface over any SQL database. You've probably used something similar in Python (psycopg2, SQLAlchemy). But for Postgres specifically, pgx/v5 is the idiomatic choice:

database/sql + lib/pqpgx/v5 directly
Postgres-specific typesManual scanning onlyNative support (UUID, JSONB, arrays)
Context cancellationSupportedSupported
Connection poolingBuilt-inpgxpool package
Batch queriesNoYes
Idiomatic for Postgres?Generic — fits any DBYes — Postgres-first

You already know Postgres well. pgx exposes it more directly than database/sql does — less abstraction, more control.

$ go get github.com/jackc/pgx/v5

Go Connecting and Running Queries

Single connection

import "github.com/jackc/pgx/v5"

conn, err := pgx.Connect(ctx, os.Getenv("DATABASE_URL"))
if err != nil {
    return fmt.Errorf("connecting to db: %w", err)
}
defer conn.Close(ctx)

Write — Exec

// Exec for INSERT/UPDATE/DELETE — no rows returned
_, err := conn.Exec(ctx,
    "INSERT INTO events (session_id, role, content) VALUES ($1, $2, $3)",
    sessionID, role, content,
)

Parameters are positional ($1, $2...), not named. This is standard Postgres — same as psycopg2.

Read one row — QueryRow

var id string
err := conn.QueryRow(ctx,
    "SELECT id FROM sessions WHERE guild_id = $1",
    guildID,
).Scan(&id)
if errors.Is(err, pgx.ErrNoRows) {
    // no session yet for this guild
}

pgx.ErrNoRows is a sentinel — the same pattern you wrote in the referee package. errors.Is unwraps the chain to find it.

Read multiple rows — Query

rows, err := conn.Query(ctx,
    "SELECT role, content FROM events WHERE session_id = $1 ORDER BY created_at",
    sessionID,
)
if err != nil { return err }
defer rows.Close()

for rows.Next() {
    var role, content string
    if err := rows.Scan(&role, &content); err != nil {
        return err
    }
    // use role, content
}
return rows.Err()
defer rows.Close() is not optional. rows holds an open server-side cursor. If you return early without closing it — including on an error path — the underlying connection is held busy until the pool recycles it. Always defer rows.Close() immediately after checking the error from Query.

Also check rows.Err() after the loop. The loop exits normally when rows are exhausted and when a network error occurs mid-stream. Only rows.Err() tells you which.

Go TDD: Build pkg/store

Same pattern as pkg/llm: define the interface in the consumer, implement with pgx, test with a stub.

The interface lives in pkg/store/store.go:

Red — store_test.go
package store

import (
    "context"
    "testing"
)

type stubStore struct{ events []Event }

func (s *stubStore) AppendEvent(_ context.Context, e Event) error {
    s.events = append(s.events, e)
    return nil
}

func (s *stubStore) RecentEvents(_ context.Context, sessionID string, limit int) ([]Event, error) {
    if len(s.events) <= limit {
        return s.events, nil
    }
    return s.events[len(s.events)-limit:], nil
}

func TestStore(t *testing.T) {
    ctx := context.Background()
    s := &stubStore{}

    events := []Event{
        {SessionID: "s1", Role: "player",   Content: "I attack the goblin"},
        {SessionID: "s1", Role: "narrator", Content: "The goblin dodges"},
        {SessionID: "s1", Role: "player",   Content: "I try again"},
    }
    for _, e := range events {
        if err := s.AppendEvent(ctx, e); err != nil {
            t.Fatal(err)
        }
    }

    t.Run("RecentEvents returns last N", func(t *testing.T) {
        got, err := s.RecentEvents(ctx, "s1", 2)
        if err != nil { t.Fatal(err) }
        if len(got) != 2 {
            t.Errorf("got %d events, want 2", len(got))
        }
        if got[0].Content != "The goblin dodges" {
            t.Errorf("unexpected first event: %q", got[0].Content)
        }
    })
}
Green — store.go
package store

import "context"

type Event struct {
    SessionID string
    Role      string // "player" | "narrator" | "referee" | "npc"
    Content   string
}

type Store interface {
    AppendEvent(ctx context.Context, e Event) error
    RecentEvents(ctx context.Context, sessionID string, limit int) ([]Event, error)
}

Then the real implementation in pkg/store/pgx_store.go — integration-tested like pkg/llm, skipped without DATABASE_URL:

type PGXStore struct{ conn *pgx.Conn }

func New(ctx context.Context, connStr string) (*PGXStore, error) {
    conn, err := pgx.Connect(ctx, connStr)
    if err != nil { return nil, fmt.Errorf("store.New: %w", err) }
    return &PGXStore{conn: conn}, nil
}

func (s *PGXStore) AppendEvent(ctx context.Context, e Event) error {
    _, err := s.conn.Exec(ctx,
        "INSERT INTO events (session_id, role, content) VALUES ($1, $2, $3)",
        e.SessionID, e.Role, e.Content,
    )
    return err
}

func (s *PGXStore) RecentEvents(ctx context.Context, sessionID string, limit int) ([]Event, error) {
    rows, err := s.conn.Query(ctx,
        "SELECT role, content FROM events WHERE session_id = $1 ORDER BY created_at DESC LIMIT $2",
        sessionID, limit,
    )
    if err != nil { return nil, err }
    defer rows.Close()

    var events []Event
    for rows.Next() {
        var e Event
        e.SessionID = sessionID
        if err := rows.Scan(&e.Role, &e.Content); err != nil { return nil, err }
        events = append(events, e)
    }
    return events, rows.Err()
}
Updated request flow with memory:
player types action
→ store.AppendEvent(player message)
→ store.RecentEvents(last 20)
→ build prompt from history + new message
→ llm.Complete → narrator response
→ store.AppendEvent(narrator response)
→ send to Discord
The Narrator now has context going back across sessions.
